# 🧠 系统设计文档：年薪百万作战系统

## 第一性原理推导

### 为什么人会半途而废？

1. **意志力有限** → 解决方案：自动化减少决策消耗
2. **反馈延迟** → 解决方案：每日可视化进度+AI即时反馈
3. **环境摩擦** → 解决方案：利用通勤/午休碎片时间，周末整块时间
4. **目标模糊** → 解决方案：拆解到每日可执行动作
5. **孤独作战** → 解决方案：公开承诺+网络监督
6. **家庭冲突** → 解决方案：早起+碎片时间，绝不占用陪娃时间

### 核心约束（不可违背）

| 约束 | 说明 | 应对策略 |
|-----|------|---------|
| 21:00下班 | 晚上无整块时间 | 通勤+午休成为主战场 |
| 22:00到家 | 陪娃到23:00 | 早起1小时+周末整块 |
| 周末陪娃 | 无法全天学习 | 早起2h+娃午睡2h+晚上1h |
| 每周≈20h | 原始容量28.5h×70%利用率 | 聚焦一条主线，提高密度，减少废动作 |

> ⚠️ 历史版本写「每周19h」但分项相加为 28.5h、百分比之和 149%，是错的。已在 [routine.md](schedule/routine.md) 修正为：原始容量 28.5h，计划目标 ≈20h（70% 利用率）。

## 聚焦原则（最高优先级元规则）

**同一时间只跑一条主线。深度复利，广度腐烂。**

- 当前唯一主线：**项目1 AI Code Review** + **算法筑基**。
- 项目2/3 已冻结（见各自文档顶部），解冻需满足显式条件。
- 后端 Go 学习降级为「项目1 需要时再学」，不单独排期。
- 任何新想法先进 `ideas/` 暂存区，不立即开新坑。

**判定标准**：如果一周内没有给主线项目提交过一次真实代码，这周就是失败的——不论刷了多少题、读了多少书。

## 目标重定义：从结果指标到可控输入

薪资是结果，不完全可控。把目标重定义为**可控的输入与产出**，薪资只是副产物。

| 维度 | ❌ 不可控结果指标 | ✅ 可控输入/产出指标 |
|-----|----------------|-------------------|
| 算法 | 面试通过 | 每周独立做出 ≥6 题 + 错题本更新 |
| 项目 | 拿到 offer | 项目1 每周 ≥2 次真实代码 commit |
| 输出 | 简历好看 | 每周 1 篇技术拆解（teach to learn） |
| 复盘 | 心态积极 | 每周日时间审计 + AI 周报完成 |
| 家庭 | — | 陪娃时间 0 侵占（红线，不可交易） |

**北极星指标**：连续 8 周达成上述输入指标 → 启动简历投递。薪资 1000k 是这些输入的滞后结果。

    ┌──────────┼──────────┐

### 核心机制

| 机制 | 原理 | 实现 |
|-----|------|------|
| **自动化检查** | 减少意志力消耗 | GitHub Actions定时检查 |
| **AI复盘** | 即时反馈+客观诊断 | 每周调用AI API分析日志 |
| **公开承诺** | 社会压力驱动 | 仓库公开+朋友监督+违约惩罚 |
| **可视化进度** | 游戏化反馈 | README动态仪表盘（`scripts/dashboard.js` 自动生成） |
| **模板化日志** | 降低记录摩擦 | 每日模板，填空式记录 |
| **教学式输出** | teach to learn，倒逼深度理解 | 每周1篇技术拆解，存 `blog/` |

### 每周拆解规则（teach to learn）

- **频率**：每周日（娃睡后 21:00-23:00 时段）产出 1 篇。
- **主题**：本周算法专题、或项目1 的一个设计决策、或一个踩坑。
- **格式**：`blog/YYYY-MM-DD-<slug>.md`，≥800 字，必须包含「为什么这么做」而非只写「怎么做」。
- **完成定义**：发到 GitHub 仓库并（可选）同步到技术社区。
- **为什么**：能讲清楚才是真懂。只刷题不输出 = 用消费冒充生产。
- **红线**：连续 2 周没产出拆解 → 下周算法量砍半，腾出时间补写。
